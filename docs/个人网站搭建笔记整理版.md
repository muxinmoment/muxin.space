# 个人网站搭建笔记整理版

这篇内容基于我的手写笔记整理而来，并补充了实际落地时需要注意的工程细节。

当前这套个人网站的核心组合是：

- Web 框架：`Astro`
- 部署平台：`Cloudflare Pages`
- 域名注册 / 管理：`阿里云`

这套方案适合我现在这种目标明确的内容型个人站：

- 主要是文章、分享、项目公开记录
- 不追求一开始就做复杂交互
- 希望部署简单、访问轻、维护成本低

---

## 1. 为什么选 Astro

在个人网站场景下，常见选择大概有三类：

- `Vue 3`
- `React`
- `Astro`

它们都能做网站，但适合的重心不一样。

### Vue / React 更适合什么

`Vue 3` 和 `React` 更适合：

- 需要大量交互的应用
- 状态很多的前端项目
- 更偏产品型、后台型、工具型的网站

如果你的站点核心是：

- 实时交互
- 复杂表单
- 用户状态
- 大量客户端逻辑

那它们会更自然。

### Astro 更适合什么

`Astro` 更适合：

- 博客
- 文档
- 知识分享站
- 内容型个人主页
- 项目展示站

因为它默认就是“内容优先、静态优先”的思路。

对我来说，当前个人网站的核心不是做一个重交互 Web App，而是做一个：

- 轻量
- 可长期更新
- 适合写内容
- 适合公开展示项目

的个人站点。

所以选择 `Astro` 是更贴合需求的，而不是单纯因为它“新”。

---

## 2. Astro 是什么

`Astro` 可以理解成一个偏内容站的现代 Web 构建框架。

它的核心理念很简单：

- 默认尽量少发 JavaScript
- 优先输出静态 HTML
- 真正需要交互的部分，再按需加客户端逻辑

这和很多传统前端框架的默认思路不一样。

很多框架是：

- 整页默认走前端应用逻辑
- 客户端要接管更多页面行为

而 `Astro` 更像是：

- 大部分内容先在构建阶段生成好
- 浏览器拿到的页面天然更轻
- 必要时再“局部激活”

所以它特别适合：

- 首屏要快
- 内容为主
- SEO 重要
- 不想一开始就背大量前端运行时成本

---

## 3. Astro 的核心架构：岛屿架构

手写笔记里提到的重点之一，就是 `Astro` 的岛屿架构（Islands Architecture）。

这个概念非常关键。

### 什么是岛屿架构

你可以把一个网页想成一整块陆地。

传统前端应用常常是：

- 整块陆地都要“通电”
- 整页都加载很多 JavaScript

而 `Astro` 的岛屿架构是：

- 大部分区域直接输出为静态 HTML
- 只有少数需要交互的部分，做成“岛屿”
- 这些小岛屿再单独加载 JavaScript

所以它的结果是：

- 静态内容更轻
- 动态能力保留
- 页面性能更好

### 为什么它适合个人站

因为个人站的大部分页面其实是：

- 文章正文
- 项目介绍
- 关于页
- 首页文本与卡片

这些内容本身不需要一开始就全部“水合”成前端应用。

真正可能需要交互的，往往只是局部：

- 搜索框
- 评论区
- 标签筛选
- 小型交互组件

那就没必要为整个页面支付统一的客户端成本。

---

## 4. 客户端岛屿

手写笔记中提到了 `client:load`，这是 Astro 常见的客户端指令之一。

### 客户端岛屿的意思

当一个组件需要在浏览器里真正运行时，我们可以通过类似：

```astro
<SomeComponent client:load />
```

让它在客户端激活。

`client:load` 的含义是：

- 页面加载后立刻在客户端执行这个组件

这类客户端岛屿适合：

- 交互必须立即可用的组件
- 页面一打开就需要工作的动态区域

比如：

- 搜索输入框
- 导航切换逻辑
- 立即可点击的互动模块

### 客户端岛屿的优势

最直接的优势是：

- 只让需要交互的地方加载 JS
- 不把整页都变成沉重的前端应用
- 对内容站来说更省资源

对个人网站尤其友好，因为你可以保持：

- 文章页很轻
- 首页很快
- 只有局部功能是动态的

---

## 5. 服务器岛屿

手写笔记里还提到了服务器岛屿，大意是把页面拆成多个可单独处理的服务端区域。

### 可以怎么理解

服务器岛屿可以理解为：

- 页面中的某些区域由服务端单独负责渲染
- 不同区域可以更灵活地决定何时生成、如何加载

你可以把它想成：

- 页面不是一整块死板的统一渲染结果
- 某些部分可以有更独立的服务端处理方式

### 它的价值

它的好处主要在于：

- 对动态内容更灵活
- 让页面中“静态部分”和“动态部分”更好拆分
- 对性能优化更细粒度

不过对当前这个个人站阶段来说，这还不是第一优先级。

因为你现在的网站主要是：

- 公开分享
- 项目介绍
- Markdown 内容

这些更偏静态内容。

所以现阶段你最应该先理解的是：

- Astro 的默认静态输出
- 岛屿架构
- 按需加载交互组件

服务器岛屿可以作为后面进阶理解。

## 6. 为什么 Astro 适合当前这个站

回到我的实际需求，这个站当前的主线不是“复杂产品”，而是：

- 知识分享
- 项目复盘
- 个人表达
- 长期内容沉淀

这意味着我真正关心的是：

- 页面够轻
- 写 Markdown 顺手
- 组件能复用
- 部署简单
- 后续扩展不痛苦

而 `Astro` 在这些点上都比较契合：

### 1. 静态输出友好

大部分页面可以直接生成静态 HTML，部署简单、性能稳定。

### 2. Markdown 内容天然友好

对知识分享站来说，Markdown 是非常舒服的内容载体。

### 3. 交互按需加载

以后如果要给局部功能加 React / Vue 组件，也可以渐进增强。

### 4. 更适合“内容主站”

比起把站点做成完整前端应用，Astro 更适合把精力放在：

- 内容结构
- 页面叙事
- 信息组织
- 长期更新节奏

## 7. 当前这套部署方案为什么顺手

手写笔记里记了三件基础设施：

- 框架：Astro
- 平台：Cloudflare
- 域名代理 / 注册：阿里云

这里稍微补充一下更准确的说法。

### Cloudflare 在这里扮演什么角色

当前这套站点里，Cloudflare 主要扮演：

- 静态站部署平台
- CDN 分发层
- 域名接入与 HTTPS 托管层

如果用的是 `Cloudflare Pages`，那它非常适合这种项目，因为：

- 构建简单
- GitHub 直连方便
- 自动部署体验好

### 阿里云在这里扮演什么角色

阿里云主要是：

- 域名注册商
- 域名管理平台

如果域名在阿里云买的，那么常见做法是：

- 域名保留在阿里云
- DNS 或 nameserver 切到 Cloudflare
- 再把网站流量接入 Cloudflare

## 8. Cloudflare Pages 部署完整流程

这里补的是实际可操作的部署流程，不只是“可以部署”，而是按当前这套项目真正能跑通的方式来。

### 第一步：本地先把项目跑通

在接 Cloudflare 之前，先确认本地项目没有问题：

```bash
npm install
npm run build
npm run check
```

如果这三步都能过，说明：

- 依赖正常
- Astro 配置正常
- 页面与内容引用正常

这一步非常重要，因为很多 Cloudflare 报错，其实源头是本地项目本来就没构建成功。

### 第二步：把仓库推到 GitHub

Cloudflare Pages 最顺手的方式是直接连 GitHub 仓库。

基础流程：

1. 本地代码提交到 Git
2. 推送到 GitHub
3. 确保主分支例如 `main` 是最新代码

### 第三步：在 Cloudflare 创建真正的 Pages 项目

这里要特别注意：

- 选择的是 `Pages`
- 不是 `Workers`
- 不是 `Workers Builds`

如果项目类型选错，后面就会走到完全不同的部署链路，容易开始出现很多看起来“像配置问题”，其实是“产品选错了”的报错。

### 第四步：连接 GitHub 仓库

在 Cloudflare `Workers & Pages` 里创建 `Pages` 项目后：

1. 选择 `Connect to Git`
2. 授权 GitHub
3. 选择当前仓库
4. 选择生产分支，一般是 `main`

### 第五步：配置构建参数

对于当前这个 Astro 静态站，推荐配置是：

- Framework preset: `Astro`
- Build command: `npm run build`
- Build output directory: `dist`
- Root directory: `/`
- Node version: `20` 或 `22`

这套配置对应你当前仓库的实际情况：

- 构建命令就是 `npm run build`
- Astro 输出目录就是 `dist`
- 项目根目录就是仓库根目录

### 第六步：首次部署

保存配置后，Cloudflare 会自动：

1. 拉取仓库
2. 安装依赖
3. 执行 `npm run build`
4. 发布 `dist`

如果成功，就会先给你一个默认的 Cloudflare 二级域名。

### 第七步：后续更新方式

后面最舒服的地方就在这里：

- 本地改代码
- 提交到 Git
- `git push`
- Cloudflare 自动重新构建并部署

也就是说，后续不需要手动上传文件，也不需要每次自己 SSH 发站点。

## 9. Cloudflare 部署踩坑总结

这部分是这次最有价值的实战经验。

### 坑 1：把 Pages 项目和 Workers 项目搞混

这是最容易踩的坑。

如果你的项目本质上是：

- Astro 静态站
- 输出 `dist`
- 没有 Worker 脚本入口

那就应该走 `Pages`。

如果误走成 `Workers` 或 `Workers Builds`，你会看到一堆很奇怪的命令配置，比如：

- `npx wrangler deploy`
- `npx wrangler versions upload`

这些命令是给 Worker 用的，不是给当前这类静态站用的。

### 坑 2：把部署命令写成 `wrangler deploy`

这个报错的典型表现是：

- 提示你在 Pages 项目里跑了 `wrangler deploy`
- 然后说缺少 Worker 入口文件

原因很简单：

- `wrangler deploy` 期待的是 Worker 脚本
- 你的站点只有静态构建结果，没有 `src/index.ts` 这种 Worker 入口

所以静态站不要走这条命令。

### 坑 3：以为一定要 `wrangler.jsonc`

如果你走的是：

- `Cloudflare Pages + Git 自动部署`

那通常不需要额外维护 `wrangler.jsonc`。

`wrangler.jsonc` 更适合：

- Worker 项目
- 命令行手动部署
- 需要显式绑定 KV / D1 / R2 / Durable Objects 的项目

对当前这个站来说，标准 Pages Git 部署已经足够了。

### 坑 4：看到 KV 报错就以为必须配 KV

你之前遇到过 KV namespace 冲突：

- Cloudflare 自动尝试创建某个 KV
- 发现同名已存在
- 然后流程中断

但这不代表“当前项目必须要 KV”。

正确判断方式是：

- 先看代码里有没有真的用到 KV
- 如果没有，就不要为了消除某条报错，提前把 KV 绑定配死

当前这个站就是典型的“不需要 KV”的项目。

### 坑 5：在构建详情页里找修改入口

Cloudflare 的“某次构建详情页”基本是只读的。

你能看到：

- 构建命令
- 部署命令
- 分支
- 状态

但这并不是主要配置入口。

真正应该改的是项目设置页里的构建设置。

## 10. 阿里云域名购买与接入流程

如果域名是在阿里云买的，推荐的使用方式是：

- 域名继续放在阿里云名下
- 通过修改 nameserver，把 DNS 接到 Cloudflare
- 再在 Cloudflare 里绑定到你的 Pages 项目

### 第一步：在阿里云购买域名

买域名时主要考虑三件事：

- 名字是否好记
- 后缀是否适合个人品牌
- 是否方便后续长期使用

对个人站来说，常见选择是：

- `.com`
- `.space`
- `.xyz`
- `.dev`

如果你是想做长期个人品牌，通常优先考虑：

- 好记
- 容易口播
- 视觉上干净

### 第二步：完成实名

在阿里云买完域名后，通常需要完成实名信息。

这一步属于域名基础管理流程，不完成的话很多后续操作会受影响。

### 第三步：把域名添加到 Cloudflare

去 Cloudflare：

1. `Add a site`
2. 输入你的域名
3. 选择免费计划即可
4. 进入 DNS 检查和 nameserver 分配流程

Cloudflare 会给你两条 nameserver，例如：

- `xxx.ns.cloudflare.com`
- `yyy.ns.cloudflare.com`

### 第四步：回阿里云修改 nameserver

在阿里云域名控制台里：

1. 打开目标域名管理页
2. 找到 DNS / 域名服务器设置
3. 把原来的阿里云 nameserver 替换成 Cloudflare 给你的两条
4. 保存

这一步的作用是：

- 把域名解析控制权交给 Cloudflare

### 第五步：等待 Cloudflare 激活域名

改完 nameserver 后，需要等一段时间同步。

Cloudflare 激活后，你的域名状态会变成可用。

### 第六步：把域名绑定到 Pages 项目

在 Cloudflare Pages 项目里：

1. 打开 `Custom domains`
2. 添加主域名，例如 `muxin.space`
3. 如果需要，再添加 `www.muxin.space`

这样网站就不再是 Cloudflare 默认二级域名，而是你自己的域名。

## 11. 域名与部署相关的几个关键理解

这里是搭站时特别容易混淆的几个概念。

### 1. 域名不等于服务器

域名只是访问入口，不是网站本体。

网站真正跑在哪里，要看部署平台：

- 你的情况是 `Cloudflare Pages`

### 2. 域名后缀不决定速度

`.com`、`.space`、`.cn` 并不会直接决定访问速度。

真正影响体验的是：

- 服务器 / CDN 节点位置
- 网络线路
- 页面本身轻不轻

### 3. 是否备案，不是看后缀，而是看部署位置

如果网站部署在中国大陆节点，通常要考虑 ICP 备案。  
如果部署在境外或 Cloudflare 这类海外网络，一般不是先看后缀，而是先看托管位置。

所以：

- 买阿里云域名 ≠ 自动变成“国内部署”
- 用 Cloudflare Pages ≠ 自动需要 ICP 备案

### 4. 当前这套方案为什么省事

当前这套路线的优点是：

- 买域名在阿里云
- 部署在 Cloudflare Pages
- 解析交给 Cloudflare
- 更新靠 Git 自动部署

这样对个人站来说，已经足够轻量、稳定、低维护。
