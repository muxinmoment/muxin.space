# 文章分类与标签规范

这份规范是给 `muxin.space` 后续所有文章用的。

目标很简单：

- 让文章结构长期稳定
- 让分类一眼能看懂
- 让 tags 不会越写越乱
- 让我以后写新文章时可以直接套模板

## 1. 内容结构规则

每篇文章都用两层结构：

- `category`：这篇文章的主类型，只能有一个
- `tags`：这篇文章涉及的主题关键词，可以有多个

一句话理解：

- `category` 解决“这篇文章本质上是什么”
- `tags` 解决“这篇文章具体在讲哪些主题”

## 2. 当前固定分类

目前站内固定使用 3 个主分类，不要随意新增。

### 项目记录

适合这些内容：

- Aido 的迭代记录
- 个人网站搭建过程
- 某个功能从想法到上线的复盘
- 部署、迁移、重构、踩坑的项目推进日志

判断标准：

如果这篇内容的核心是“我这个项目做了什么、怎么推进的、为什么这么改”，就归到 `项目记录`。

### 技术分享

适合这些内容：

- Astro / React / FastAPI / Go / Cloudflare 这类技术拆解
- 工程方案讲解
- 某个知识点的教程化说明
- 工具链、配置、实现路径的总结

判断标准：

如果这篇内容的核心是“把一个技术问题讲清楚，让别人可以复现”，就归到 `技术分享`。

### 思考随笔

适合这些内容：

- AI 边界、学习方法、个人判断
- 创作状态、关系反思、表达方式
- 不偏教程，但值得公开沉淀的思考

判断标准：

如果这篇内容的核心是“观点、经验、反思”，就归到 `思考随笔`。

## 3. tags 使用规则

### 3.1 基本规则

- 每篇文章建议 `3-5` 个 tags
- 能少就少，不要为了显得专业硬塞很多
- 统一使用小写英文或 `kebab-case`
- tags 尽量写主题词，不写空泛情绪词

推荐风格：

- `astro`
- `cloudflare`
- `fastapi`
- `agent-loop`
- `deploy`
- `personal-site`

不推荐风格：

- `我的感悟`
- `一些记录`
- `随便写写`
- `技术相关`

### 3.2 tags 的 4 种来源

后面打 tags 时，优先从下面 4 种角度选：

#### 技术栈标签

例如：

- `astro`
- `react`
- `fastapi`
- `postgresql`
- `cloudflare`

#### 项目主题标签

例如：

- `aido`
- `muxin-space`
- `kacha-ai`
- `personal-site`

#### 问题域标签

例如：

- `deploy`
- `streaming`
- `agent-loop`
- `state-management`
- `workflow`

#### 内容语境标签

例如：

- `reflection`
- `journal`
- `notes`

## 4. 实际打标签建议

每篇文章优先遵循这个顺序：

1. 先定 `category`
2. 再选 `1-2` 个技术 / 项目 tags
3. 再补 `1-2` 个问题域 tags
4. 如果确实需要，再补 `1` 个内容语境 tag

这样写出来的 tags 会比较稳定，不会散。

## 5. 现有文章参考

### 建站文章

- `category`: `技术分享`
- `tags`: `astro`, `cloudflare`, `personal-site`, `deploy`, `notes`

### 开站记录

- `category`: `项目记录`
- `tags`: `site`, `journal`, `astro`

### AI 情绪边界文章

- `category`: `思考随笔`
- `tags`: `ai`, `reflection`, `emotion`, `thinking`

## 6. 以后写新文章时直接套这个模板

```md
---
title: "这里替换标题"
description: "这里写一句话摘要，解释这篇文章为什么值得点开。"
pubDate: 2026-06-08
updatedDate: 2026-06-08
category: "技术分享"
tags:
  - astro
  - deploy
  - personal-site
draft: true
---
```

## 7. 最后判断口诀

如果你写到一半还在纠结怎么分类，就用这个口诀：

- 在讲“项目怎么推进” -> `项目记录`
- 在讲“技术怎么实现” -> `技术分享`
- 在讲“我怎么理解这件事” -> `思考随笔`

如果你写 tags 时开始想堆很多词，就停一下，只保留：

- 最核心的技术 / 项目
- 最核心的问题域
- 最必要的内容语境
